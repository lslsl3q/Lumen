# 思维簇编写规范 (Thinking Cluster Specification)

> 本规范定义了 Lumen 思维簇模块的书写标准、设计原则和最佳实践。
> 任何人（包括 AI）都可以按此规范编写新的思维模块，放入对应簇目录即可生效。

---

## 一、基本格式

每个思维模块是一个 `.txt` 文件，放在 `thinking_clusters/<簇名>/` 目录下。

### 必填部分（四段）

```
【思考模块：模块名称】
简短描述这个模块做什么（一句话）

【触发条件】
什么情况下这个模块应该被激活。
用自然语言描述场景，因为系统通过向量语义匹配来选择模块。

【核心功能】
这个模块具体做什么。
描述要清晰具体，但不要写实现代码。

【执行流程】
步骤一：xxx
步骤二：xxx
...
按顺序列出思考步骤。
```

### 选填部分

```
【注意事项】
边界条件、容易出错的地方、特殊规则。

【示例】
具体的输入输出示例，帮助理解模块行为。
```

## 二、设计原则

### 1. 单一职责
一个模块只做**一种**思维动作。不要把"意图分类 + 约束确认 + 结果验证"塞进一个文件。

反例：
```
【思考模块：万能助手】
先判断意图，再确认约束，然后推理，最后验证...
```

正例：
```
【思考模块：意图分类与策略选择】
（只做意图判断和策略匹配）
```

### 2. 语义自描述
模块的文件名和内容要包含**充分的关键词**，让向量检索能准确匹配。

反例：
```
【思考模块：think】
Do the right thing based on context.
```
（关键词不足，向量检索难以命中）

正例：
```
【思考模块：逻辑链脆弱性分析】
当需要验证推理结论的可靠性时，检查论证链条中的薄弱环节...
```
（"逻辑链"、"脆弱性"、"验证"、"推理"、"可靠性"都是有效锚点词）

### 3. 簇内正交
同一个簇内的模块应该覆盖**不同角度**，不要重复。

例如 `reasoning_patterns/` 簇：
- `analytical.txt` — 结构化分析推理
- `creative.txt` — 创意发散与收敛
- `inductive.txt` — 归纳总结推理

这三个模块解决不同类型的推理需求，互补不重叠。

### 4. 簇间互补
不同簇解决思维链的**不同阶段**，形成完整管道。

```
前思考阶段 → task_context 簇（认知准备）
推理阶段  → reasoning_patterns 簇（思维执行）
后思考阶段 → self_correction 簇（质量校验）
```

## 三、思维类型分类

### 认知型（Cognitive）
**目的**：理解任务、准备上下文
**典型模块**：
- 意图分类
- 约束条件确认
- 上下文构建
- 边界界定

**适用阶段**：前思考（任务开始前）

### 推理型（Reasoning）
**目的**：执行思维过程、产出结论
**典型模块**：
- 结构化分析推理
- 创意发散与收敛
- 归纳总结
- 演绎推理
- 类比推理

**适用阶段**：推理阶段（核心思维过程）

### 校验型（Validation）
**目的**：检查结论质量、发现错误
**典型模块**：
- 结果自检与修正
- 逻辑一致性验证
- 事实性核查
- 反例搜索

**适用阶段**：后思考（输出前最后一道关）

### 元认知型（Meta-cognitive）
**目的**：反思思维过程本身、调整策略
**典型模块**：
- 策略选择与切换
- 思维疲劳检测
- 退阶决策（何时降级）
- 思路总结与复盘

**适用阶段**：贯穿全程，或作为链的最后一环

## 四、思考阶段与簇的映射

### 推荐的三阶段链

```json
{
  "steps": [
    {"cluster": "task_context", "top_k": 2, "min_score": 0.25},
    {"cluster": "reasoning_patterns", "top_k": 2, "min_score": 0.3},
    {"cluster": "self_correction", "top_k": 1, "min_score": 0.35}
  ]
}
```

**为什么是这个顺序**：
1. **先准备**（task_context）：了解任务全貌、确认约束
2. **再执行**（reasoning_patterns）：选择并应用推理方法
3. **后检查**（self_correction）：验证结果、修正错误

### 向量融合的作用

每步检索后，系统会将查询向量与结果向量融合（默认 0.8:0.2）：
- 0.8 的原始查询权重 → 保持核心意图不被淹没
- 0.2 的结果权重 → 让后续簇能"感知"前一步找到了什么

**利用方式**：如果你想让后续簇能根据前一步结果调整行为，可以在模块内容中写"根据前一步的判断结果..."。

### 降级模式

如果某个簇搜索为空（没有模块达到 min_score 阈值），该簇被标记为 `degraded` 并跳过，管道继续执行。

**设计建议**：
- min_score 不要设太高（推荐 0.25-0.35）
- 每个簇至少有 3-5 个模块，覆盖足够多的关键词
- 核心簇（如 reasoning_patterns）多放几个模块

## 五、簇间衔接规范

### 输出契约
模块不产出结构化数据，它产出的是**自然语言指导**，被注入到 AI 的上下文中。模块之间不直接传递数据。

### 依赖关系
模块之间应该**无硬依赖**。每个模块应该是自包含的指导文本，即使单独被检索到也能独立生效。

反例（有依赖）：
```
根据上一步"意图分类"的结果，如果是"分析型"则执行以下步骤...
```
（如果上一步降级了，这个模块就无效了）

正例（无依赖）：
```
当你需要分析复杂问题时，将问题拆解为独立的子维度...
```
（无论前一步是什么，这个指导都有价值）

### 信息衔接方式
通过**向量融合**隐式传递信息。前一步检索到的模块内容会影响查询向量，使后续步骤更容易命中相关的模块。这是自然的语义衔接，不需要手动编码。

## 六、编写实战指南

### Step 1：确定模块属于哪个簇
- 准备阶段的思考 → `task_context`
- 推理方法 → `reasoning_patterns`
- 结果检查 → `self_correction`
- 如果现有簇不合适 → 新建簇，并在 chains.json 中添加

### Step 2：写模块内容
遵循四段必填格式。关键是【触发条件】要写充分的关键词。

### Step 3：测试
1. 把 .txt 文件放入对应簇目录
2. 重启后端（或等待自动重新索引）
3. 发送与模块内容相关的消息
4. 在 `/tokens` 调试面板中检查"思维簇"层是否出现了你的模块

### Step 4：调优
- 如果模块没被检索到：增加【触发条件】中的关键词
- 如果不相关的模块被检索到：提高 chains.json 中的 min_score
- 如果 token 超预算：精简模块内容，或调高 token_budget

## 七、示例对比

### 好的模块

```
【思考模块：意图分类与策略选择】

【触发条件】
收到用户输入后，需要判断意图类型：是分析、创作、问答、执行还是情感表达。
当用户的问题可以有多种解读方向时，此模块帮助选择最合适的回应策略。

【核心功能】
将用户输入归类为五种意图类型，并为每种类型选择对应的处理策略。
不暴露分类过程，自然地执行策略。

【执行流程】
步骤一：识别关键词和语气，判断意图类型
步骤二：如果意图不明确，优先按"问答型"处理
步骤三：根据意图类型选择对应策略，在后续回复中贯彻

【注意事项】
复合意图需要组合多种策略
不要在回复中暴露你做了意图分类
```

**为什么好**：
- 触发条件包含丰富关键词（意图、分类、策略、分析、创作、问答）
- 核心功能清晰具体
- 执行流程可操作
- 注意事项补充了边界情况

### 差的模块

```
好好想想再回答。
```

**为什么差**：
- 没有格式结构
- 触发条件缺失（向量检索无法匹配）
- 执行流程不具体
- 没有实际指导价值

## 八、扩展指南

### 新建簇
1. 在 `thinking_clusters/` 下创建新目录（英文名，下划线分隔）
2. 放入至少 3 个 .txt 模块
3. 在 `chains.json` 中添加新的链配置，或修改现有链的 steps

### 新建链
在 `chains.json` 的 `chains` 对象中添加新条目：

```json
{
  "chains": {
    "default": { ... },
    "creative_writing": {
      "name": "creative_writing",
      "steps": [
        {"cluster": "narrative_structure", "top_k": 2, "min_score": 0.25},
        {"cluster": "character_voice", "top_k": 3, "min_score": 0.3},
        {"cluster": "prose_style", "top_k": 2, "min_score": 0.3}
      ],
      "token_budget": 800,
      "fusion_weight_query": 0.75,
      "fusion_weight_results": 0.25
    }
  }
}
```

在角色 JSON 中指定使用哪条链：
```json
{
  "thinking_cluster_chain": "creative_writing"
}
```

### 自由组合
- 簇的数量不限
- 簇的顺序自由
- 每个簇可以放任意数量的模块
- 不同角色可以用不同的链
