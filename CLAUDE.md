# Lumen 项目工作指南

> 本文件每次对话全量加载，所有规则必须遵守。

---

## 项目身份

Lumen 是一个 AI 桌面应用，目标是专业级产品。

- **后端**：Python 3.11 + FastAPI（全异步架构：AsyncOpenAI）
- **前端**：Tauri 2 + React + TypeScript（桌面应用）
- **通信**：SSE 流式（后端 → 前端）
- **数据**：SQLite（history.db）

---

## 架构铁律：按系统角色分层

不按功能分文件夹，按系统角色分。单向依赖，不反向。符合Claude Code架构哲学。例如：
```
lumen/
  config.py       # 全局配置（单例）
  core/           # 大脑 — 决策循环、会话状态
  tools/          # 双手 — 每个工具一个 .py
  services/       # 神经 — 基础设施（LLM、存储、记忆、搜索）
  prompt/         # 嘴巴 — 提示词拼接、角色管理、上下文注入
  types/          # 词汇 — 类型定义、常量
```

### 新功能放哪

| 新建的东西 | 放在 | 例 |
|-----------|------|---|
| AI 可调用的能力 | `tools/xxx.py` | 搜索、生图、计算 |
| 数据存储/检索引擎 | `services/xxx.py` | 向量库、记忆存储 |
| 对话流程/编排逻辑 | `core/xxx.py` | Agent循环、梦系统 |
| 提示词拼接/上下文注入 | `prompt/xxx.py` | 世界书注入、情感描述 |
| 类型定义/常量 | `types/xxx.py` | 消息格式、枚举 |
| HTTP API 端点 | `api/routes/xxx.py` | 新资源 CRUD |
| 前端 API 请求 | `src/api/xxx.ts` | chat.ts |
| 前端状态管理 | `src/hooks/useXxx.ts` | useChat.ts |
| 前端 UI 组件 | `src/components/` | ChatInterface.tsx |

前端和后端同一套思想：组件 → hooks → api，不反向依赖。

## 执行规则

### 开发环境

- **包管理器**：用 `uv pip install`，不用 `pip`（虚拟环境在 `.venv/`，没有 pip 模块）
- **代理**：下载依赖需要代理 `http://127.0.0.1:7897`
  ```bash
  uv pip install --proxy http://127.0.0.1:7897 <包名>
  ```
- **LLM 代理**：本地 `127.0.0.1:4000` 转发
- **架构**：全链路异步（config.py → llm.py → chat.py → memory.py → API 层），不改回同步

### 工作流程

1. **新会话先读** `CODE_INDEX.md` 了解项目结构
2. **改完代码后检查**：是否新增/删除/改变了文件职责？是 → 更新 CODE_INDEX.md
3. **写代码标准**：单一职责、低耦合、配置分离。重复 2 次以上 → 提取函数
4. **用中文思考和沟通**

### 类型系统规则

- **全局类型**（跨 3+ 模块）→ `lumen/types/`，按领域拆文件（messages、events、tools）
- **模块类型**（1-2 个模块用）→ 就近放 `模块/types.py`（如 `prompt/types.py`、`services/types.py`）
- **边界用 Pydantic**：LLM 输出解析、数据库读取、HTTP 请求/响应、JSON 配置文件 — 外部数据进门时验证
- **内部用 TypedDict**：模块间消息传递、SSE 事件构造 — 零开销，IDE 能提示
- **类型文件只放类型定义**和紧耦合的构造/判断函数，不放业务逻辑
- **模块拆分/合并/新增时**，照此规则分配类型归属

### 隐私规则（处理 .gitignore 时）

需要忽略文件时，先判断隐私级别：
- **公开规则**（放 `.gitignore`）：Python缓存、虚拟环境、IDE配置、数据库、日志等标准文件
- **私密规则**（放 `.git/info/exclude`）：Claude记忆、AI报告、内部索引、个人笔记等

不要默认全放 `.gitignore`，先判断再问用户确认。

## 沟通风格

- **教学模式**：不只写代码，要教会用户理解代码。解释做了什么、为什么这么做
- **用中文**：思考、分析、解释都用中文
- **用比喻**：帮助用户理解复杂概念

## 信息保存判断规则

新信息产生时，按以下顺序判断放哪：

| 判断问题 | 是 → 放到 |
|---------|----------|
| "忘了这个会出bug或破坏架构吗？" | **CLAUDE.md**（本文件） |
| "这是未来要做的事吗？" | **memory/roadmap.md**（短期或长期） |
| "这是用户告诉我做事方式的纠正？" | **memory/feedback_xxx.md** |
| "这是告诉我去哪找东西的指针？" | **memory/reference_xxx.md** |
| "这是项目当前状态或决策？" | **memory/current_status.md** |

## Roadmap 维护规则

**完成短期任务后**：
1. 在 roadmap.md 标记完成
2. 扫描长期规划，判断是否有条目可以提升到短期（依赖满足、难度适中）
3. 有则提升并告诉用户："XXX 现在可以做了，已加入短期规划"

**产生新想法时**：
1. 评估实现难度和前置依赖
2. 依赖满足、难度适中 → 短期规划
3. 依赖不足、难度较大 → 长期规划

**完成里程碑后**：重新审视整个长期区，重新评估优先级。
